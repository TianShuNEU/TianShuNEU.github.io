Managing Thread

Basic thread management

Launching a thread

Starting a new thread using the c++ Thread Library always boils down to constructing a std::thread object,and specify a task to run on that thread.

A task is a callable type,including function,functor,and lambda expression.

  1. Function 

    void do_sth();
std::thread my_thread(do_sth);

  2. Functor: A functor is a class that has overrided function call operator. 

     1:    class do_sth
 2: {
 3: public:
 4:     void operator ()()
 5:     {
 6:         do_sth_func();
 7:         do_sth_func_else();
 8:     }
 9: };
10: do_sth ds;
11: std::thread my_thread(ds);

    In this case,the supplied functor is copied into the storage belonging to the newly created thread of execution and invoked from there.when pass a temporary rather than a named variable,you must avoid "c++'s most vexing parse":if you write code like this 

    std::thread my_thread(do_sth());

    compiler will interpret it as a function declaration,you should write this instead: 

    std::thread my_thread((do_sth()));
std::thread my_thread{do_sth()};

  3. Lambda expression This is a new feature from c++11,allowing you to write local function. 

    std::thread my_thread([](){do_sth();do_sth_else();});

The task is invoked since it is specified to the thread object,after that,you need to explicitly decide whether to wait for it to finish(by joing with it),or leave it run on its own(by detaching it).If you don't decide before the object is destroyed,then std::thread destructor calls std::terminate(),and your program is terminated.

Waiting for a thread to complete

Just call join() function is sufficient, it is simple and brute force,if you need more fine-grained control over waiting for a thread,such as to check whether a thread is finished or to wait a certain period of time,you will need condition variables and futures.

If you start a thread within a function,and has access to the local variables in that function(although it is a bad idea),you should guarantee the thread to finish before the function exits by calling join().

Waiting in exceptional circumstances

If you need to wait for the thread to finish,you should pick carefully the place in the code where you call join(),this means that the call join() is liable to be skipped if an exception is thrown after the thread has been started but before the call to jion().

One way to ensure join() is called before the thread object is destroyed is to use the standard Resource Acauisition Is Initialization(RAII)idiom and provides a class that does the join in its destructor: 

 1: class ThreadGuard
 2: {
 3: public:
 4:     ThreadGuard(const std::thread& _t):t{_t}{};
 5:     ~ThreadGuard(){
 6:         if(t.joinable())
 7:             {
 8:                 t.join();
 9:             }
10:     };
11:     ThreadGuard(const ThreadGuard&other)=delete ;
12:     ThreadGuard& operator=(const ThreadGuard&right)=delete;
13: private:
14:     std::thread&t;
15: };
16: 
17: struct func{
18:     int&i
19:     func(int &i_):i{i_}{}
20:     void operator()()
21:     {
22:         for(unsigned j{0};j<20;++j)
23:             {
24:                 do_sth(i);
25:             }
26:     }
27: };
28: 
29: void f
30:     {
31:      int some_local_state{0};
32:      func my_func(some_local_state);
33:      std::thread t(my_func);
34:      ThreadGuard g(t);
35: 
36:      do_some_thing_in_current_thread();
37:     }

when the execution of the current thread reaches the end of f(),the local objects are destroyed in reverse order of construction.Consequently,the ThreadGuard g is destroyed first,and join() is called even if do_some_thing_in_current_thread() throws an exception.

If you don't need to wait for a thread to finish,you can avoid this exception-safety issue by detaching it.

Running thread in the background

If you are detaching a thread,you can usually call detach() immediately after the thread has been started.After the call detach completes,this breaks the association of the thread with the std::thread object and ensures that std::terminate() won't be called when the std::thread object is destroyed,even though the thread is still running in the background.

Detached thread are often called daemon thread that runs in the background without explicit user interface,such threads are typically long-running,

Passing args to a thread function

Although other mechanisms could be used to pass args to a thread function(function object,args can be it's data member),the Thread Library provides you with an easy way of doing it.

Pass by value

Passing args to the callable object or function is foundamentally as simple as passing additional args to std::thread constructor, but std::thread constructor does't know the what kind of arguments the function needs,whether by value or by reference,but bear in mind that by default the args are copied into internal storage(by value). 

 1: #include <iostream>
 2: #include <string>
 3: #include <thread>
 4: void fun(int i, std::string str)
 5: {
 6:     str+=" world";
 7:     ++i;
 8: }
 9: 
10: int main(int argc, char *argv[])
11: {
12:     int i{9};
13:     std::string str{"Hello"};
14:     std::cout<<"Before thread,i:"<<i<<" str:"<<str<<std::endl;
15:     std::thread t(fun,i,str);
16:     t.join();
17:     std::cout<<"After thread,i:"<<i<<" str:"<<str<<std::endl;
18: 
19:     return 0;
20: }
21: 

Before thread,i:9 str:Hello
After thread,i:9 str:Hello

Pass by reference

If the function in the thread passes arguments by reference,you should use std::ref. 

 1: #include <iostream>
 2: #include <string>
 3: #include <thread>
 4: #include <functional> //std::ref
 5: void fun(int  &i, std::string&str)
 6: {
 7:     str+=" world";
 8:     ++i;
 9: }
10: 
11: int main(int argc, char *argv[])
12: {
13:     int i{9};
14:     std::string str{"Hello"};
15:     std::cout<<"Before thread,i:"<<i<<" str:"<<str<<std::endl;
16:     std::thread t(fun,std::ref(i),std::ref(str));
17:     t.join();
18:     std::cout<<"After thread,i:"<<i<<" str:"<<str<<std::endl;
19: 
20:     return 0;
21: }
22: 

Before thread,i:9 str:Hello
After thread,i:10 str:Hello world

Pass a member function

You can also pass a member function pointer as the function,provided you supply a suitable object pointer as the first arguments. 

 1: #include <iostream>
 2: #include <thread>
 3: #include <functional>
 4: #include <string>
 5: class X
 6: {
 7: public:
 8:     void test_func(int i,std::string& str)
 9:         {
10:             ++i;
11:             str+=" word";
12:         }
13: };
14: int main(int argc, char *argv[])
15: {
16:     X x;
17:     int i{9};
18:     std::string str{"Hello"};
19:     std::thread t(&X::test_func,&x,i,std::ref(str));
20:     t.join();
21:     std::cout << "After thread,i = " <<i<<" str = " <<str<<"\n";
22:     return 0;
23: }
24: 

After thread,i = 9 str = Hello word

Args can't be copied,but only moved

Some type in c++ can't be copied,such as std::unique_ptr,Only one such pointer can point to a given object at a time,and when the pointer is destroyed,the pointed-to object is deleted.A std::unique_ptr can only be moved: the data held within the object is transferred over to another,leaving the original object "empty".

Where the source object is temporary,the move is automatic,but where the source is source is a named value,the transfer must be required directly by invoking std::move(). 

void process_big_objcet(std::unique_ptr<big_object> );
std::unique_ptr<big_object> p{new big_object};
std::thread t(process_big_object,std::move(p));

Transferring ownership of a thread

A std::thread object is responsible for managing a thread of execution,instances of std::thread are movable not copyable.This ensures that only one object is associate with a particular thread of execution at any one time.
